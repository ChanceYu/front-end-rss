# 滥用...拓展运算符导致崩掉，被老板娘约谈了3小时

深夜11点半，办公室里只剩下我们前端组几个人，屏幕的光映着一张张疲惫的脸。

线上报警群突然炸了——新上线的配置中心页面，用户一加载就白屏，数据量稍大的企业客户几乎全军覆没。查来查去，性能分析工具最终指向一行我曾引以为傲、写得无比顺手的代码：

```
const mergedData = { ...defaultConfig, ...userConfig };
```
正是这行“优雅”的 `...`，这个被我们当作肌肉记忆的拓展运算符，在遇到一个包含数万条嵌套规则的用户配置时，成了拖垮整个页面的元凶。

那个夜晚，我们不得不紧急回滚、重写合并逻辑，折腾到凌晨才勉强平息。这次惨痛的教训让我彻底明白：**拓展运算符不是银弹，滥用它，就是在代码里埋雷。**

---

### 简洁的假象背后：浅拷贝的幽灵

用 `...` 或 `Object.assign()` 合并对象，意图清晰，代码漂亮：

```
const defaults = { theme: 'dark', version: '1.0' };
const userConfig = { theme: 'light', showTips: true };
const finalConfig = { ...defaults, ...userConfig };
```
但当对象嵌套时，这份“简洁”便开始滋生副作用。它们进行的都是**浅拷贝**，只复制了第一层属性的引用。

```
const source = {
  user: 'Alice',
  profile: { age: 25, hobbies: ['coding', 'reading'] }
};

const merged = { ...source, user: 'Bob' };
merged.profile.age = 30; // 你以为只在改新对象？

console.log(source.profile.age); // 输出：30！原始数据被污染了。
```
在一个复杂的应用里，这种悄无声息的数据污染如同幽灵，引发的 Bug 往往难以追踪。你以为在操作一个新对象，实则可能撼动了整个应用状态的基石。

---

### 被忽视的性能炸弹：当数据量爆炸时

浅拷贝的危害尚且属于逻辑隐患，而更直接、更具破坏性的问题，则发生在大数据量场景下。这正是让我们团队加班到深夜的罪魁祸首。

**拓展运算符在处理大型数组或深层嵌套的复杂对象时，性能开销会呈指数级增长。** 它需要遍历所有可枚举属性，并将其展开到一个新对象中。当你在合并一个包含数万条记录、每条记录又有复杂子结构的配置对象时，这个过程会阻塞主线程，直接导致页面渲染卡顿甚至崩溃。

```
// 假设这是从后端获取的庞大用户配置
const hugeUserConfig = fetchHugeConfig(); // 一个深度很大或属性极多的对象

// 危险操作：试图用拓展符合并
const combined = {
  ...defaultSettings, // 默认配置
  ...hugeUserConfig   // 海量数据在此展开，可能导致长时间停顿或内存暴涨
};
```
在我们的案例中，正是这样的操作，让浏览器在解析和合并阶段消耗了远超预期的时间和内存，最终导致页面无法响应。

---

### 如何安全地“合并”：深拷贝与策略选择

那么，如何避免踩坑？关键在于根据场景，选择正确的工具。

**深拷贝是隔离副作用的前提**

对于需要完全隔离的嵌套数据，应首先进行深拷贝。现代浏览器和 Node.js 环境提供了原生的 `structuredClone()` 方法，它使用结构化克隆算法，安全且高效。

```
const source = { profile: { age: 25 } };

// 先深拷贝，再合并
const safeMerged = { ...structuredClone(source), user: 'Bob' };
safeMerged.profile.age = 30;
console.log(source.profile.age); // 输出：25 ✅ 原始数据安全
```
**针对大数据量的合并策略**

当处理庞大数据时，应避免在内存中进行一次性全量展开和合并。

- **分治策略**：如果可能，将大配置拆分成多个模块，按需加载和合并。
- **惰性合并**：只在访问某个具体属性时，才计算最终的合并值。
- **使用专用工具库**：对于极其复杂的合并逻辑（如深度合并、自定义规则等），考虑使用 `lodash.merge` 等经过充分优化的库，但同样需警惕其性能开销。
- **源头优化**：与后端协商，减少不必要的数据传输，或在服务端完成部分合并逻辑。
