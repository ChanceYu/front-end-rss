# 【早阅】技术债治理手册：五步作战计划

前 Monzo CTO Meri Williams 阐述了如何通过 “五步作战法”（可视化、基准对比、全员宣讲、争取资源掩护、全力交付），在不中断业务交付的前提下，将复杂的 “单体怪兽” 逐步拆解，从而显著降低认知负荷并提升 4.5 倍的研发吞吐量。

#### 一、技术债务的定义与财务类比

本次技术分享聚焦于如何有效处理技术债务。演讲者拥有从硬件黑客到大型企业（如宝洁）SAP 系统管理，再到构建英国政府网站 gov.uk 的丰富经验，并在多家金融科技公司担任过首席技术官。技术债务的定义涵盖了任何使代码难以理解、难以扩展或难以修改的因素。这种代码状态常常引发维护者的沮丧感，因为他们必须花费大量精力来解析前人留下的结构。

> 永远要记住，维护你代码的下一个人可能是一个杀人狂，也可能就是你自己。

技术债务的产生机制与金融债务类似，都是因为当前需要某物或想要某物，但无力立即支付全部成本。这种借贷行为本身并非邪恶或愚蠢，但当 “利息支付” 变得过高时，问题便出现了。因此，保持对技术债务利息成本的清晰认知至关重要，必须避免陷入 “用信用卡支付抵押贷款” 的境地。

#### 二、上下文、开发体验与案例研究

##### 理解历史决策的上下文

在评估现有代码时，不应轻易假设前任人员是愚蠢的。由于缺乏当时所面临的压力和背景信息，很容易产生这种轻率的判断。因此，架构决策记录（ADRs）被认为是极具价值的工具，它们提供了决策背后的 “为什么” 和上下文。当上下文发生变化时，可以重新评估决策，同时理解其历史合理性。

许多工程师不得不处理比自己年龄还老的系统，有些系统甚至达到了 “古董” 级别，代码可能已不可用，需要多重备份来保障核心业务连续性。对开发人员体验（DX）而言，技术债务最糟糕的影响体现在三个方面：流程状态（Flow state）、认知负荷（Cognitive load）和反馈循环（Feedback loops）。

##### 案例研究：Damos 单体应用

在 Danish 金融科技公司 Pleo，一个被称为 “Damos” 的单体应用是技术债务的集中体现。由于服务命名采用了神话而非合理的名称，导致了入职困难。该应用在当时为公司创造了收入，但已变得难以理解、结构混乱、文档缺失，并且在客户量达到 30,000 时，系统扩展性面临巨大压力，导致了频繁的 P1 事件，甚至引发了是否需要实施 “变更冻结” 的讨论。

#### 三、解决技术债务的五步策略

尽管重写系统诱人，但重写往往耗时过长并可能扼杀公司。因此，采取了不同的渐进式方法，即一个五步计划来处理技术债务。这个计划旨在逐步解耦系统，而不是一次性推倒重来。该策略的实施确保了在解决问题的同时，公司仍能保持业务的正常运转和新功能的交付。

- 可视化问题
- 利用 DORA 指标（如周期时间）向董事会展示现状。当时他们的周期时间长达 5.6 天，远低于行业标准
- 与行业基准进行对比
- 通过对比 “精英团队” 的指标，让非技术背景的董事会成员意识到问题的严重性
- 进行宣传和解释
- 向全公司（包括销售团队）解释 Damos 是如何阻碍客户价值交付的
- 争取适度的外部支持（Air Cover）
- Meri 争取到了 50% 的时间用于修复 Damos，同时承诺继续交付新功能。她甚至对 CEO 许下 “极端问责” 的承诺：如果一年后我还在抱怨它，就把我解雇
- 交付成果
- 实施 API 优先的架构变革，对团队进行大规模培训，并逐步从单体中剥离功能

##### 可视化、对标与争取支持

可视化阶段大量使用了《加速》（Accelerate）一书中的 DORA 指标，例如周期时间（Cycle Time）。即使团队试图 “优化” 指标，其结果也是将工作拆分成更小、风险更低的批次。通过对标行业精英表现，清晰地向董事会展示了当前周期时间（5.6 天）与理想状态的巨大差距，即使他们不完全理解 DORA 指标，也明白了情况的严重性。

> 我真诚地告诉他们，如果一年后我还在抱怨 Damos，就解雇我。

争取 “空中掩护” 意味着要求允许团队将 50% 的时间用于修复 Damos，同时继续交付新功能。此举体现了极高的问责制。执行层面，团队转向了 API 优先的架构变更，并与一家名为 Skiller Whale 的公司合作，对工程师进行 API 设计和测试的全面培训，因为许多工程师甚至不理解 API 的价值。



| 行动类别 | 具体措施 | 目标 / 结果 |
| --- | --- | --- |
| 架构方向 | 转向 API 优先 | 实现系统解耦 |
| 能力建设 | 与 Skiller Whale 进行 API 培训 | 提升工程师设计和测试 API 的能力 |
| 执行成果 | 将 Damos 模块化为 API | 降低了关键业务流程的风险（红线区域） |


#### 四、日常维护与领导者责任

经过三年的努力，周期时间改进了 40%，吞吐量提升了 450%。最重要的是，团队没有完全隐藏起来只处理技术债务，而是一直在交付客户价值。完全停止功能开发会带来业务风险，因此持续交付是偿还债务过程中必须坚持的原则。

##### 日常技术债务管理实践

技术债务并非不可避免，但可以通过小步快跑的方式进行管理，避免等待其演变成需要大规模重写的地步。解决开发体验的三个痛点是关键：缺少反馈循环时添加测试；认知负荷过高时重构复杂代码（可使用 CRAP 指标进行静态分析）；缺少流程状态时增加文档。

- 每次修改代码时，优先改进该区域的技术债务。
- 构建和重构相结合，让平台自然演化。
- 为所有临时性的 “脏黑客” 设置明确的过期日期（SLA/TODO）。

对于领导者而言，关键信息是必须主动为团队创造空间来处理技术债务，并明确告知团队可以优先处理债务问题。这遵循了 “童子军规则”：每次离开时，都让代码比发现时更干净。此外，领导者有时需要充当 “挡箭牌”，在董事会或高层面前坚持认为当前做法是错误的，必须改进才能实现更快的交付速度。

#### 早读洞察

1、技术债务的明确定义：技术债务被定义为任何使代码难以理解、难以扩展或难以修改的代码结构或实践，这直接影响了系统的长期健康和可维护性。

2、技术债务与财务债务的类比：技术债务的产生与财务借贷相似，关键在于管理由此产生的 “利息”—— 即维护和变更代码所需付出的额外努力和时间成本。

3、架构决策记录的重要性：架构决策记录（ADRs）是记录历史决策背景的关键工具，帮助团队理解过去的选择，避免重复犯错，并为未来重评估提供依据。

4、开发体验的三大核心要素：最糟糕的技术债务会同时影响开发人员的流程状态、认知负荷和反馈循环，这三者是衡量开发体验的关键指标。

5、避免大规模重写系统：大规模重写往往耗时是预期的两倍，并可能导致公司停滞或失败；应采取渐进式、API 优先的解耦策略来逐步偿还债务。

6、量化技术债务的行业对标：利用 DORA 指标（如周期时间）可视化当前性能，并与行业基准进行对比，能够有效地向非技术决策者证明问题的严重性。

7、争取资源与问责机制：成功处理技术债务需要争取到高层支持（如分配 50% 时间），并设立明确的问责期限，以确保项目不会无限期拖延。

8、持续交付是债务清理的前提：在偿还技术债务的同时，必须持续交付客户价值；完全停止新功能开发会带来业务风险，因此需要平衡投入。

9、日常维护遵循童子军规则：技术债务管理应融入日常工作，遵循 “童子军规则”，即每次修改代码时都应使其比发现时更完善。

10、领导者需为改进创造空间：领导者的核心职责是明确授权，并为团队提供处理技术债务所需的时间和精力，确保改进工作能够落地并带来交付收益。
