# 【第3662期】快速上手 Claude 技能：从零到可用

前言

作为前端开发者，你是否希望 AI 能记住你的工作流程、生成符合团队风格的文档，甚至自动处理多步骤任务？Claude 技能正是解决方案。本指南从零开始，手把手教你构建第一个技能，让你只需一次设置，就能在每次对话中自动应用你的偏好、流程和最佳实践，节省时间、减少重复操作，快速提升工作效率。

如果你曾希望 Claude 能记住你的工作流程、遵循团队的风格指南，或在无需每次重新解释的情况下执行多步骤操作 —— 技能就是答案。

技能是一个简单的文件夹，你上传到 Claude 后，它会教 Claude 如何处理特定任务，并在每次对话中保持一致。你无需每次都从零指导 Claude，只需教一次，它就会在相关场景中自动应用这些知识。

#### 技能的强大之处

技能最适合处理可重复的工作：生成符合品牌风格的文档、进行冲刺计划会议、引导新客户，或在 Notion、Linear、Figma 等工具之间管理多步骤工作流。

当技能与 MCP（模型上下文协议）集成结合时，效果尤其显著。可以这样理解：

> MCP 是厨房，技能是食谱。

MCP 让 Claude 能访问你的工具和数据。技能则告诉 Claude 如何使用这些工具 —— 正确的顺序、检查步骤和输出格式。没有技能，用户连接 MCP 集成后只会看到空白界面；有了技能，工作流程会自动启动。

#### 技能结构

每个技能都是一个文件夹，可能包含以下四类内容：

```
 your-skill-name/
 ├── SKILL.md        ← 必需。技能核心内容。
 ├── scripts/        ← 可选。Python、Bash 等脚本。
 ├── references/     ← 可选。文档、指南、示例。
 └── assets/         ← 可选。模板、字体、图标。
```
严格规则：主文件必须命名为 `SKILL.md`（区分大小写，不能有其他变体）。文件夹名称必须使用 kebab-case（如 `my-skill-name`，不能写成 `My Skill Name`）。

#### Claude 如何决定使用技能

Claude 不会一次性加载所有技能，而是采用智能的三级系统：

- Level 1 — YAML 前置信息：始终存在于 Claude 的系统提示中，用于判断技能是否相关。
- Level 2 — SKILL.md 内容：完整指令，仅在 Claude 判断技能适用时加载。
- Level 3 — 关联文件：额外参考资料，仅在需要时加载。

这意味着最关键的是前置信息描述。如果 Claude 无法从描述中理解何时使用技能，其余内容就无效。

#### 构建你的第一个技能

**步骤 1：先明确使用场景**

在写任何代码前，先搞清楚：

- 用户希望完成什么目标？
- 需要哪些多步骤流程？
- 涉及哪些领域知识或工具？

一个好的使用场景示例：

```
 使用场景：冲刺计划
 触发条件：用户说“帮我规划这个冲刺”
 步骤：获取项目状态 → 分析容量 → 建议优先级 → 在 Linear 创建任务
 结果：冲刺计划完成，无需手动设置
```
**步骤 2：编写 YAML 前置信息**

这是整个技能中最关键的部分。最小必需格式如下：

```
 ---
 name: your-skill-name
 description: 它能做什么。当用户说 [具体短语] 时触发。
 ---
```
描述必须做到两点：说明技能功能，并明确触发条件。包含用户可能说出的真实短语。长度不超过 1024 个字符，禁止使用 XML `< >` 标签。

#### 有效技能与无效技能的区别

```
 # ✅ 有效 — 具体，包含真实触发短语
 description: 分析 Figma 设计文件并生成开发交接文档。当用户上传 .fig 文件，或说 “design specs”“component documentation”“design-to-code handoff” 时使用。

 # ❌ 无效 — 太模糊，没有触发条件
 description: 帮助处理项目。
```
#### 步骤 3：编写指令

在前置信息之后，用普通 Markdown 写指令，结构如下：

```
 # 技能名称

 ## 指令说明
 ### 步骤 1: [操作]
 具体、可执行的描述。不要写“验证数据”，而是写：
 “运行 python scripts/validate.py --input {filename}”。

 ## 示例
 用户说: "创建新项目"
 1. 通过 MCP 获取现有项目
 2. 使用提供参数创建新项目
 结果: 项目创建完成，并提供确认链接

 ## 故障排查
 错误: 连接被拒绝
 原因: MCP 服务器未运行
 解决: 设置 > 扩展 > 重新连接
```
保持主 `SKILL.md` 文件在 5,000 字以内。详细文档可放入 `references/` 文件夹，并链接调用 ——Claude 仅在需要时加载。

#### 三类技能（附真实示例）

**类别 1：文档与资源创建**

用于生成一致、高质量的输出 —— 报告、演示文稿、UI 设计、代码。

Anthropic 库中的真实示例：`frontend-design`、`docx`、`pptx`、`xlsx` 技能。  
可嵌入风格指南、输出模板和质量检查表，无需外部工具。

Anthropic 技能库：https://github.com/anthropics/skills/tree/main/skills

**类别 2：工作流自动化**

用于需要每次都保持一致方法的多步骤流程。真实示例：`skill-creator` 技能 —— 交互式引导用户构建新技能。可嵌入有序步骤、阶段间验证点和迭代优化循环。

skill-creator：https://github.com/anthropics/skills/blob/main/skills/skill-creator/SKILL.md

**类别 3：MCP 增强**

将原始工具访问转化为可靠、可引导的工作流程。真实示例：Sentry 的 `sentry-code-review` 技能 —— 使用 Sentry 错误监控数据通过 MCP 自动分析并修复 GitHub PR 中的 bug。

可嵌入领域专业知识、按顺序调用 MCP 并处理用户本应自己处理的错误。

#### 五种真正有效的模式

##### 模式 1：顺序工作流编排

适用：顺序重要的多步骤流程

```
 创建账户 → 设置支付 → 创建订阅 → 发送欢迎邮件
```
关键技巧：明确步骤顺序、步骤间依赖关系、失败回滚指令。

##### 模式 2：多 MCP 协调

适用：跨多个服务的工作流

```
 Figma MCP: 导出资源
 Drive MCP: 上传并整理
 Linear MCP: 创建开发任务
 Slack MCP: 发布交接摘要
```
关键技巧：阶段清晰分离、服务间数据明确传递、集中错误处理。

##### 模式 3：迭代优化

适用：输出可通过复审不断改进

```
 草稿 → 验证 → 优化 → 再验证 → 完成
```
关键技巧：明确质量标准、验证脚本、定义停止条件。

##### 模式 4：上下文感知工具选择

适用：同一目标，根据上下文选择不同工具

```
 文件 > 10MB?        → Cloud Storage MCP
 协作文档?          → Notion MCP
 代码文件?          → GitHub MCP
```
关键技巧：明确决策标准、回退选项、向用户透明说明工具选择原因。

##### 模式 5：领域专属智能

适用：添加超出工具访问本身的专业知识

金融合规示例：

```
 检查制裁名单 → 核实管辖权 → 风险评估
 → 处理支付 → 记录完整审计日志
```
关键技巧：操作前的合规检查、嵌入领域规则、完整文档记录。

#### 如何测试技能

1、触发测试 — 运行 10–20 条应该激活技能的查询。目标：90% 自动激活。同时确认不会误触发无关话题。

> 最佳调试技巧：直接问 Claude——“何时会使用 \[技能名\] 技能？” 它会引用描述，缺口立即显现。

2、功能测试 — 确认输出正确、API 调用成功、边缘情况处理到位。

3、性能对比 — 测量前后差异：



| 指标 | 无技能 | 有技能 |
| --- | --- | --- |
| 交互次数 | 用户每次都需提供指令，共 15 次往返对话 | 自动执行工作流，仅需 2 次澄清提问 |
| API 稳定性 | 3 次 API 调用失败并需重试 | 0 次 API 调用失败 |
| 消耗 token | 消耗 12,000 个 token | 消耗 6,000 个 token |


**提升性能的核心原理**

- 渐进式披露（Progressive Disclosure）：技能采用三层披露系统（YAML 元数据、SKILL.md 正文、关联文件）。这种设计能确保只有在相关时才加载详细指令，从而最大限度地减少 Token 消耗并保持专业水准。

**量化成功标准**

- 触发率：理想目标是技能在 90% 的相关查询中自动触发，无需用户手动开启。
- 效率提升：通过减少完成工作流所需的工具调用次数和总 Token 数来衡量性能。
- 减少执行错误：通过在指令中嵌入最佳实践和错误处理（如重试逻辑或验证脚本），可以实现工作流中 0 次 API 调用失败的目标

#### 出现问题时的解决方案



| 症状 | 解决方法 |
| --- | --- |
| 上传时 “找不到 SKILL.md” | 重命名为 SKILL.md（区分大小写） |
| “Invalid frontmatter” 错误 | 检查 --- 分隔符，关闭所有引号 |
| 技能从不自动加载 | 在描述中加入真实触发短语 |
| 技能对无关查询触发 | 在描述中加入 “Do NOT use for...” |
| MCP 调用失败 | 先单独测试 MCP，检查工具名大小写 |
| Claude 忽略指令 | 使用 CRITICAL: 标题，将关键步骤放前面 |
| 响应慢或性能下降 | 将文档移入 references/，保持 SKILL.md < 5,000 字 |


#### 分发技能

1、个人使用：压缩文件夹 → Claude.ai → 设置 → 功能 → 技能 → 上传。

2、组织使用：管理员可在整个工作区部署技能，实现自动更新和集中管理。

3、社区共享：托管在 GitHub。README 放在仓库根目录，不要放在技能文件夹里。外部描述技能时，以成果为先：

> ✅ “几秒钟内完成完整项目工作区设置，而不是 30 分钟”
> 
> ❌ “一个包含 YAML 前置信息和 Markdown 指令的文件夹”

#### 上传前检查清单

- 文件夹名称使用 kebab-case
- 主文件命名为 SKILL.md（完全匹配，区分大小写）
- YAML 前置信息上下用 `---` 包裹
- name 字段：使用 kebab-case，无空格，无大写字母
- description 字段同时说明技能功能 和 触发条件
- 文件中禁止使用 XML `< >` 标签
- 指令具体明确，不模糊
- 包含错误处理
- 在直接触发短语和同义触发短语上测试
- 确认技能不会在无关查询上误触发

#### 15 分钟构建你的第一个技能

最快速从零到可用技能的步骤：

1、打开 Claude.ai，说：“使用 `skill-creator` 技能帮我为 \[你的使用场景\] 创建一个技能”  
2、描述你最常用的 2–3 个工作流程  
3、让 `skill-creator` 生成带正确前置信息的 `SKILL.md`  
4、上传并运行触发测试  
5、根据触发情况迭代调整

大多数人按照这个流程，15–30 分钟内就能得到一个可用技能。

#### 实例：构建 “UI 架构师” 技能

我们将构建一个名为 ui-architect-pro 的技能，旨在将设计稿精准转化为生产级 React 组件。

1、文件夹结构

创建一个符合 kebab-case 命名的文件夹（严禁空格或大写）：

```
 ui-architect-pro/
 ├── SKILL.md              # 核心：定义“什么时候触发”与“如何执行”
 ├── references/           # 深度知识：设计系统变量、组件库规范
 └── assets/               # 资源：基础代码模板、图标
```
架构思考：这种结构确保了技能的可移植性（Portability）。无论是在 Claude.ai、Claude Code 还是通过 API 调用，它都能保持行为一致。注意： 文件夹内不得包含 README.md，以免干扰 Claude 的扫描。

2、编写 SKILL.md

```
 ---
 name: ui-architect-pro
 description: 分析 UI 设计稿并生成生产级 React 组件。当用户要求“生成代码”、“转换设计图”或上传图片/Figma 链接时触发。
 ---

 # UI 架构师指令

 ## 执行流程

 1、规格解析：识别用户提供的 UI 标注、布局和交互态。
 2、规范对齐：关键必须参考 `references/brand-tokens.md` 中的 CSS 变量，确保色值符合标准。
 3、代码生成：使用 React + Tailwind 架构。若涉及基础组件，参考 `references/component-lib.md`。
 4、质量自检：输出前，对照 `references/qa-check.md` 进行性能审查。

 ## 示例 (Examples)

 用户说：“帮我把这张登录页设计图写成代码。”

 行动：解析布局 -> 调用品牌变量 -> 生成响应式组件。
```
**深度解析：为什么这样设计？**

- 触发条件（What + When）：在 description 中，我们不仅写了 “它能做什么”，还明确了触发短语（如 “上传图片”）。这能确保你的技能在 90% 的相关查询中自动触发。
- 行动导向（Actionable）：指令必须具体。与其说 “请验证数据”，不如说 “参考 references/qa-check.md 进行审查”。

3、利用 `references/` 优化性能

将详尽的品牌变量存放在 references/brand-tokens.md 中。

这是性能优化的精髓。为了保持响应极速，SKILL.md 建议控制在 5,000 字以内。将琐碎的规范移入参考文件夹，能让 Claude 在平时保持 “清醒”，仅在特定步骤才 “翻书”。

#### 总结

技能就是可复用的专业知识。你只需一次性将工作流程偏好、领域知识和最佳实践编码到 Claude 中，每次对话都会自动受益。

投入成本很小：一份写得清晰的 `SKILL.md` 收益巨大：输出一致、无需重复解释、减少修正，AI 助手真正按照你的方式工作。

从一个使用场景开始。一个工作流程开始。发布、测试、优化。
